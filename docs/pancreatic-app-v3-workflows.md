# 胰腺癌临床数据收集 APP v3 工作流方案

## 本轮新增要求

- U 盘导出的数据必须能导入另一台电脑上的同款 APP。
- 自选 Excel 导出以“每行一个患者、每列一个数据种类”为主表。
- 输入入院时间和出院时间后自动计算住院时间。
- 病理和影像图片数据默认不保存，只保存报告内容和结构化摘要。
- 化验和检查可以选择全部每一次，也可以选择入院首次、最后一次、术前最近一次等时间规则。
- 内置本地大模型聊天窗口，用于软件使用帮助和字段解释。
- 继续用多 agent 审查优化整体方案。

## 双端离线数据闭环

临床电脑端负责采集和确认数据，分析电脑端负责导入、复核、分析和归档。两端都安装同款 APP，但角色不同：

- 临床电脑 APP：采集 HIS/LIS/PACS/病案系统里的文字和报告，保存本地 SQLite，生成 U 盘导出包。
- U 盘导出包：承载人看的 Excel、APP 导入用的 `data/*.csv` 标准机器表、manifest、checksum 和导出报告。
- 分析电脑 APP：读取 U 盘导出包，校验 SHA-256，预览字段和病例数，导入 staging，去重后进入分析库。

导出包不能只靠手工复制裸 Excel。每次导出都必须有批次号 `export_id`，另一台电脑导入时按 `export_id + record_uuid + content_hash` 去重。Excel 是给人看，另一台 APP 导入时优先读取 `data/` 目录下的标准化 CSV。

推荐导出包结构：

```text
ClinicalDataExport_{医院代码}_{设备ID}_{YYYYMMDD_HHMMSS}_{export_id}/
  export_manifest.json
  checksum.sha256
  export_report.txt
  excel/export.xlsx
  data/patient_master.csv
  data/encounter.csv
  data/diagnosis.csv
  data/lab_report.csv
  data/lab_result.csv
  data/report_record.csv
  data/treatment.csv
  data/followup.csv
  dict/field_dictionary.csv
  dict/unit_dictionary.csv
  evidence/
```

## 自选 Excel 导出

默认导出的 `患者主表` 必须是：

- 每行一个患者。
- 每列一个数据种类。
- 第一列为脱敏研究编号。
- 入院时间、出院时间、住院天数可同时导出。
- 多诊断可导出为主诊断列、全部诊断合并列，也可另放 `诊断明细` sheet。
- 多次化验和检查默认不把全部明细塞进主表，而是按用户选择的时间规则展开关键列。

如果同一患者有多次住院，`患者主表` 的入院时间、出院时间和住院天数必须来自一个明确的“索引住院次”。导出向导提供选择：首次胰腺癌相关住院、手术住院、最近一次住院、或手动指定住院次。完整住院次另放 `住院次明细` sheet。

化验和检查时间规则：

- 每一次：进入长表，一行一次结果或一条报告。
- 入院首次：取入院后最早一次。
- 最后一次：取当前就诊或筛选范围内最晚一次。
- 术前最近一次：取手术日期前最近一次。
- 术后第一次：取手术日期后最早一次。
- 指定日期范围：保留范围内全部明细，必要时也可生成宽表。

宽表列名必须带单位，例如：

- `白蛋白_入院首次 (g/L)`
- `CA19-9_最后一次 (U/mL)`
- `血红蛋白_术前最近一次 (g/L)`

## 住院天数自动计算

用户输入或 AI 抽取到 `入院时间`、`出院时间` 后，APP 自动计算 `住院天数`。

规则：

- 如果只有入院时间，没有出院时间，住院天数显示为 `待出院`。
- 如果入院和出院都是日期，默认 `出院日期 - 入院日期 + 1`。
- 如果记录精确到时间，可同时保存小时差，但 Excel 默认导出天数。
- 如果出院时间早于入院时间，立即标红并禁止确认。
- 自动计算字段必须在字段字典里标记为 `derived`，并在导出说明页写明计算规则。
- 派生字段保存 `length_of_stay_calc_method`、`length_of_stay_source`、`length_of_stay_status`，不允许用户静默覆盖计算结果。

## 影像和病理报告策略

第一版不保存影像图片、DICOM、病理切片图片或大体照片。只保存：

- 报告类型。
- 报告日期。
- 报告标题。
- 报告原文。
- AI 抽取候选摘要。
- 人工确认后的结构化字段。
- 来源系统和来源定位。

这样能降低存储压力，也减少 U 盘转移的大文件风险。后续如果必须追溯图片，APP 只记录 PACS/病理系统报告号或检查号，不复制原图。

## 化验截图导入策略

临床电脑上可以上传 HIS/LIS 化验截图，但截图只作为本机临时预览和 OCR 输入，不进入正式数据库和 U 盘导出包。APP 只保存：

- OCR 后的报告文字。
- AI/规则解析出的化验候选字段。
- 人工确认后的结构化化验长表记录。
- 来源截图文件名、识别时间和来源文本片段。

当前 HTML/JS 原型用“浏览器原型模拟 OCR”跑通交互闭环：上传截图后生成候选，人工确认后写入 `lab_result`。正式 Windows 桌面版应接入离线 OCR 引擎，例如 Windows OCR、PaddleOCR 或 Tesseract；识别后进入候选区，必须人工确认后才写入 `lab_result`。

## 内置 AI 聊天窗口

界面建议右下角常驻“AI 助手”按钮，点击后打开聊天窗口。

右侧 AI 工作区建议做成三页签：

- `AI候选字段`：显示从报告文本抽取出的候选字段、来源、置信度、确认按钮。
- `AI助手聊天`：回答软件使用、字段填写、导出导入流程问题。
- `来源追溯`：显示当前字段来自哪段 HIS/LIS/PACS/病理报告原文。

AI 助手用途：

- 问软件怎么用：如何导入 U 盘包、如何导出 Excel、如何筛选多诊断。
- 问字段怎么填：住院天数怎么算、化验单位如何处理、长表宽表区别。
- 当前页面质控：提示哪些字段缺失、哪些 AI 候选项还没确认。
- 文本抽取解释：说明某个候选字段来自哪段报告原文。

限制：

- 不能联网。
- 不能上传患者信息。
- 不能自动诊断。
- 不能越过人工确认直接改数据库。
- 低内存电脑上限制上下文长度和聊天记录数量。
- 批量导出、导入、OCR 时暂停模型推理，避免普通 i5 电脑卡死。

## 开源本地模型后端

第一版不要把“大模型”做成唯一依赖。推荐做成三层可切换后端：

- `规则助手`：默认启用，本地字段字典、质控规则和软件帮助知识库即可运行，适合医院低配电脑。
- `本地小模型`：可选增强，用 `llama.cpp + GGUF` 在 Windows 离线调用模型文件，用于软件问答、字段解释和候选抽取。
- `开发调试`：开发电脑可用 Ollama 快速比较提示词和模型效果，正式医院离线版仍应固定模型文件和推理后端。

模型候选先按“免费开源、可离线、CPU 可跑、中文能用、许可证清楚”筛选，不直接追求最大参数量。低配 i5 / 8GB 电脑优先考虑规则助手或 0.6B 量级；i5 / 16GB 可尝试 1.7B-4B 量化 GGUF。候选包括：

- Qwen/Qwen3-0.6B、Qwen/Qwen3-1.7B-GGUF、Qwen/Qwen3-4B-GGUF。
- ggml-org/gemma-4-E2B-it-GGUF、ggml-org/gemma-4-E4B-it-GGUF。
- 其他医院允许离线部署、许可证清楚、可固定版本和 hash 的模型。

正式部署包必须记录：

- 模型文件名、版本、来源链接和许可证。
- 模型文件 SHA-256。
- 推理后端版本。
- 默认上下文长度、最大输出长度和并发限制。
- 是否启用 OCR / 抽取 / 聊天功能。

AI 模型边界必须写入界面和说明书：只做软件帮助、字段口径解释、候选字段抽取和缺失项提示；不能自动诊断、不能自动治疗决策、不能绕过人工确认写入正式库。
